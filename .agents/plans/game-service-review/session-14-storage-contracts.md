## Findings

### Finding 1: contracts_campaign_participant_invite_character.go Bundles 4 Entities
- **Severity**: medium
- **Location**: `storage/contracts_campaign_participant_invite_character.go` (195 lines)
- **Issue**: One contract file defines Reader and Store interfaces for 4 entities: Campaign, Participant, Invite, and Character. While 195 lines is not excessive, the bundling means a contributor looking for character storage contracts must search a file named for 4 entities. Other contract files are scoped more narrowly.
- **Recommendation**: Split into per-entity files: `contracts_campaign.go`, `contracts_participant.go`, `contracts_invite.go`, `contracts_character.go`. Each file defines the record type, Reader interface, Store interface, and Page type for one entity.

### Finding 2: Storage Records Import Domain Types — Acceptable Coupling Direction
- **Severity**: low
- **Location**: `storage/contracts_campaign_participant_invite_character.go:8-12`
- **Issue**: `CampaignRecord` uses `campaign.Status`, `campaign.GmMode`, etc. and `ParticipantRecord` uses `participant.Role`, `participant.Controller`. This creates a dependency from storage → domain. In strict DDD, storage should use its own types. However, in this codebase, the storage contracts are thin projection records — they're read models that reference domain enums for type safety.
- **Recommendation**: The current approach is pragmatic. Domain enums (Status, Role, Kind) are stable value types. Converting between domain and storage types for enums would add boilerplate without benefit. Accept as-is.

### Finding 3: Reader/Store Interface Split Is Consistent
- **Severity**: info
- **Location**: `storage/contracts_*.go`
- **Issue**: Every entity follows the same pattern: `XxxReader` (read-only) and `XxxStore` (embeds Reader + mutations). This allows read-only consumers to depend only on the Reader interface, following the interface segregation principle.
- **Recommendation**: Clean and consistent pattern.

### Finding 4: models.go at 575 Lines Is Generated — Not a Concern
- **Severity**: info
- **Location**: `storage/sqlite/db/models.go`
- **Issue**: This file is generated by sqlc from SQL queries. Generated code doesn't need review for maintainability — it's rebuilt from the schema.
- **Recommendation**: No review needed. Ensure `//go:generate` or equivalent is documented.

### Finding 5: ErrNotFound Sentinel Usage
- **Severity**: info
- **Location**: `storage/storage.go`
- **Issue**: `ErrNotFound` is the shared sentinel for missing records. All storage implementations return it, and callers use `errors.Is` to detect not-found conditions.
- **Recommendation**: Consistent sentinel usage.

### Finding 6: Migration Structure
- **Severity**: info
- **Location**: `storage/sqlite/migrations/`
- **Issue**: Migrations are sequential SQL files applied at startup. Schema evolution is forward-only.
- **Recommendation**: Standard approach. Consider documenting migration testing in the contributor guide.
